# Правила для AI-ассистента

## 1. Инициализация нового проекта (BOOTSTRAP)

При создании нового проекта или при первом запросе в пустом/новом проекте — АВТОМАТИЧЕСКИ выполни следующее:

### 1.1. Создай структуру папок:
```
project-root/
  .cursor/rules/        # Правила Cursor AI
  docs/                 # Документация
    features/           # Описание фич
    setup/              # Инструкции по настройке
    deploy/             # Деплой и инфраструктура
    architecture/       # Архитектурные решения
    client/             # Документация для заказчика (простым языком)
  scripts/              # Служебные скрипты
    deploy/             # Скрипты деплоя
```

### 1.2. Создай файлы `.cursor/rules/`:

**security.mdc** (alwaysApply: true):
- Все эндпоинты ЗАКРЫТЫ по умолчанию; публичные — только по явному запросу с комментарием
- Auth и авторизация на уровне middleware/guards, до бизнес-логики
- Валидация и санитизация всех входных данных
- Rate-limit на чувствительных эндпоинтах
- Секреты ТОЛЬКО в `.env`, НИКОГДА в коде или .md
- НЕ логировать токены, пароли, персональные данные

**git.mdc** (alwaysApply: true):
- НИКОГДА не делать `git push` без явного подтверждения пользователя
- Перед commit — краткий чек-лист изменений
- Формат коммитов: `[type] Описание` (feat, fix, docs, refactor, test, chore)
- Ветки: main, dev, feature/*, bugfix/*
- Деплой — только с разрешения пользователя

**docker.mdc** (globs: Dockerfile*, docker-compose*):
- Dockerfile + docker-compose.yml с первого дня
- Health checks, named volumes, restart: unless-stopped
- .env для конфигурации, .env.example как шаблон
- Multi-stage builds для production
- Все секреты через environment блок, не хардкод

**architecture.mdc** (alwaysApply: true):
- Модульный монолит: чёткие границы модулей, независимые друг от друга
- Каждый модуль: контроллеры, сервисы, репозитории, DTO, интерфейсы
- Модули общаются через контракты/интерфейсы, не напрямую
- Shared-модуль для общих утилит, типов, middleware
- Один сервис = одна ответственность
- Готовность к выделению модулей в микросервисы без рефакторинга

### 1.3. Создай стартовые файлы:
- `README.md` — описание проекта, инструкция по запуску
- `.env.example` — шаблон переменных окружения (без реальных ключей)
- `.gitignore` — стандартный для стека проекта
- `.dockerignore`
- `Dockerfile` — под стек проекта
- `docker-compose.yml` — БД + основной сервис + healthcheck
- `docs/client/README.md` — шаблон: «Эта папка содержит инструкции для пользователей продукта, написанные простым языком без IT-терминов»

### 1.4. Спроси у пользователя:
- Какой стек? (язык, фреймворк, БД) — если не указан
- Нужны ли специфичные правила для стека? — создай дополнительный .mdc

## 2. Стек по умолчанию

Если пользователь не указал стек, используй:
* **Frontend:** React + TypeScript
* **Backend:** Nest.js + TypeScript
* **Database:** PostgreSQL
* **ORM:** TypeORM

### Backend (Nest.js):
* Паттерны: Module, Controller, Service, Repository
* DTO с `class-validator` и `class-transformer`
* TypeORM c `@InjectRepository()`
* async/await, try...catch или Exception Filters
* Декораторы: `@Get()`, `@Post()`, `@Body()`, `@Param()`, `@UseGuards()`

### Frontend (React):
* Только функциональные компоненты
* Хуки: useState, useEffect, useContext, useCallback
* Типизация всех props через interface/type
* HTTP-запросы через axios или fetch

### TypeScript:
* `interface` для формы объектов, `type` для остального
* Тип `any` запрещён — используй точный тип или `unknown`

### PostgreSQL:
* Синтаксис PostgreSQL для SQL-запросов
* Корректные типы Postgres в схемах: jsonb, uuid, timestamptz

## 3. Безопасность эндпоинтов

* ВСЕ эндпоинты ЗАКРЫТЫ по умолчанию. Публичных нет, если пользователь явно не попросит.
* Каждый route/endpoint — auth и авторизация ДО бизнес-логики.
* Защита на уровне middleware/guards/декораторов (на роутере), не внутри хендлера.
* Публичный эндпоинт (webhook, health check) — комментарий с причиной.
* Валидация и санитизация всех входных данных (body, query, params).
* Rate-limit на login, registration, password reset.

## 4. Docker и контейнеризация

* Любой проект с первого дня — `Dockerfile` + `docker-compose.yml`.
* Все сервисы в одном docker-compose.yml.
* `.env` для конфигурации, `.env.example` как шаблон.
* Health checks для зависимых сервисов.
* Named volumes для персистентных данных.
* Multi-stage builds для production.

## 5. Архитектура: модульный монолит

* Чёткие границы модулей, каждый независим.
* Модуль содержит: контроллеры, сервисы, репозитории, DTO, интерфейсы.
* Общение между модулями — только через контракты/интерфейсы.
* Shared-модуль для утилит, типов, middleware.
* Один сервис = одна ответственность.
* Готовность к выделению в микросервисы без рефакторинга.

## 6. Организация файлов

* .md-файлы запрещены в корне, кроме `README.md` и `CHANGELOG.md`. Всё остальное — в `docs/`.
* Скрипты (.sh, .py утилиты) запрещены в корне. Всё — в `scripts/`, деплой — в `scripts/deploy/`.
* Структура docs/:
  - `docs/features/` — фичи
  - `docs/setup/` — настройка
  - `docs/deploy/` — деплой
  - `docs/architecture/` — архитектура
  - `docs/client/` — для заказчика (простой язык, без IT-терминов)
* Документация в `docs/client/` — на русском, простым языком, понятным людям без IT-опыта.

## 7. Авто-обнаружение правил

* Когда выявляется повторяющийся паттерн или конвенция — **предложи** создать правило в `.cursor/rules/` (.mdc с YAML frontmatter).
* Создавать только после подтверждения пользователя.
* Одно правило — один файл, одна тема, до 50 строк.
* Формат: «Вижу повторяющийся паттерн [описание]. Предлагаю зафиксировать в `.cursor/rules/[имя].mdc`. Создать?»

## 8. Git и деплой

* НИКОГДА не делать `git push` без явного текстового подтверждения.
* Перед commit — краткий чек-лист изменений.
* Никогда не деплоить без прямого разрешения.
* Пароли и доступы — ТОЛЬКО в `.env`. Никогда в .md или коде.
